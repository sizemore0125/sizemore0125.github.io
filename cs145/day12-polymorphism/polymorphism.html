Polymorphism

In Object Oriented Programming, Polymorphism is a feature that allows us to use a single type entity (method, operator or object) to represent different types in different scenarios. The word "polymorphism" comes from Greek words "poly" meaning many and "morphos" meaning forms, hence many forms.

Upcasting and Downcasting

Upcasting is the process of treating a subclass object as if it were an instance of its superclass or parent class. This happens automatically in Java because a subclass is a specialized version of its superclass, and every subclass object is inherently a superclass object.

Downcasting is the process of treating a superclass reference as if it were a reference to a subclass or child class.. This requires an explicit cast and should only be performed when it is certain that the object being referred to is an instance of the subclass.

Upcasting

Upcasting is commonly used to write generic and reusable code, as it allows you to refer to different subclasses using a single superclass type.

Let's look at an example:

public class Main {
	public static void main(String[] args) {
		PlayableCharacter logan = new PlayableCharacter("Logan", 9000, -1);
		Rogue rogue = new Rogue("Aragorn", 5, 80, 100); 
		Bard bard = new Bard("Tom Bombadil", 10, 120, "Lyre");

		display(logan);
		display(rogue);
		display(bard);
	}

	public static void display(PlayableCharacter character) {
		character.attack();
	}
}

This will display:

Logan attacks!
Aragorn performs a sneaky backstab!
Tom Bombadil sings a magical tune to weaken enemies!

In this example, we pass in instances of a PlayableCharacter, Rogue, and Bard into the display method, and this is all ok because both the Rogue and the Bard are also of type PlayableCharacter. In other words, a Rogue is a PlayableCharacter. A Bard is a PlayableCharacter. Both the Bard and Rogue are implicitly upcasted.

You can also upcast an object while instantiating that object. For example:

PlayableCharacter rogue = new Rogue("Aragorn", 5, 80, 100);

In this instance, the Java compiler will understand the rogue object as a instance of the PlayableCharacter class, not the Rogue class, meaning you only have access to fields that are defined in the PlayableCharacter class.

Downcasting

Downcasting is used to access subclass-specific methods or properties that are not available in the superclass.

Let's say that I want to use the hide method within the display method of my Main class, but the hide that is only defined in the Rogue class. To do this, I can explicitly downcast the PlayableCharacter reference character into a Rogue reference.

For example:

public class Main {
	public static void main(String[] args) {
		Rogue rogue = new Rogue("Aragorn", 5, 80, 100); 
		display(rogue);
	}

	public static void display(PlayableCharacter character) {
		character.attack();
		((Rogue) character).hide();
	}
}

Which will display:

Aragorn performs a sneaky backstab!
Aragorn hides in the shadows with stealth level 100.

There is a problem with this implementation. Now, if any other type of PlayableCharacter other than a Rogue is passed as an argument to the display method, then there will be an error. A object of type Rogue is a PlayableCharacter, but not all PlayableCharacters are of type Rogue.

To fix this, we can use the instanceof keyword in Java to check if a object is an instance of a specific subclass. For example:

public class Main {
	public static void main(String[] args) {
		PlayableCharacter logan = new PlayableCharacter("Logan", 9000, -1);
		Rogue rogue = new Rogue("Aragorn", 5, 80, 100); 
		Bard bard = new Bard("Tom Bombadil", 10, 120, "Lyre");

		display(logan);
		display(rogue);
		display(bard);
	}

	public static void display(PlayableCharacter character) {
		character.attack();
		if (character instanceof Rogue) {
			((Rogue) character).hide();
		}
	}
}

Now, this will display:

Logan attacks!
Aragorn performs a sneaky backstab!
Aragorn hides in the shadows with stealth level 100.
Tom Bombadil sings a magical tune to weaken enemies!

Key points:

  • Upcasting is always safe and implicit.
  • Downcasting requires an explicit cast and should be used cautiously.
  • Use the instanceof operator to check the object type before performing a downcast to avoid ClassCastException.

Comparing Objects

Before we introduced methods implemented by the Object class in Java, including toString, equals, and hashCode. It is common practice to override these methods when it fits your overall design intention for your program. In fact, if you override equals, it is best practice to also override hashCode. Let's looks at how we can best override these methods for our PlayableCharacter class:

public class PlayableCharacter {
    protected String name;
    protected int level;
    protected int health;

    public PlayableCharacter(String name, int level, int health) {
        this.name = name;
        this.level = level;
        this.health = health;
    }

	@Override
	public String toString() {
		return "Name: " + name + 
			   "\nLevel: " + level + 
			   "\nHealth: " + health + "\n";
	}

	@Override
	public boolean equals(Object obj) {
		if (this == obj) {
			return true;
		}

		if (!(obj instanceof PlayableCharacter)) {
			return false;
		}

		PlayableCharacter other = (PlayableCharacter)obj;
		return (
			this.name.equals(other.name) && 
			this.level == other.level && 
			this.health == other.health
		);
	}

	@Override
	public int hashCode() {
		return Objects.hash(name, level, health);
	}

    public void attack() {
        System.out.println(name + " attacks!");
    }
}

Abstract Classes

An abstract class in Java is a class that cannot be instantiated directly. Instead, it is designed to be a blueprint for other classes. Abstract classes and can include abstract methods (methods without a body) as well as concrete methods (methods with a body). Abstract classes are used to define shared behavior and enforce structure in subclasses.

For example, our PlayableCharacter class makes more sense as an abstract class, because it represents the abstract concept of a playable character. We probably never want to have a playable character in our game if it is not a specific character type like a rogue, wizard, or bard. So, lets change our PlayableCharacter class to an abstract class.

An abstract class is declared using the abstract keyword:

public abstract class PlayableCharacter {
    protected String name;
    protected int level;
    protected int health;

Then, we can make the attack method in PlayableCharacter an abstract method. We do this using the abstract keyword:

public abstract void attack();

Now, any class that extends the PlayableCharacter class must implement an attach method, or it will result in a compilation error.

Abstract classes allow to enforce structure in your code and it improves reusability, as you can define shared behavior for subclasses.

Coupling

Coupling refers to the degree of dependency between components or classes in software. It is a measure of how closely connected the classes or modules are and how much they rely on each other to function. You can never get rid of coupling, as we want to create a system of components that work together to accomplish a task, but we can reduce coupling.

Tight Coupling

Tight coupling is when classes or components are heavily dependent on each other's internal details. With tight coupling, when you change one class, it often require changes in the other, leading to reduced flexibility and maintainability.

For example:

class Bard {
	public void attack() {
		System.out.println(name + " sings a magical tune to weaken enemies!");
	}
}

class Game {
	private Bard character = new Bard();

	public void startGame() {
		character.attack();
	}
}

Here, the Game class is tightly coupled to the Bard class because it directly creates and manages a specific Bard instance. If the behavior or type of Bard changes, or if you want to have a new type of playable character, the Game class might also require changes, limiting flexibility.

Loose Coupling

Loose coupling occurs when classes or components interact through well-defined abstractions. This reduces dependency and allows components to evolve independently.

For example:

abstract class PlayableCharacter {
	abstract void attack();
}

public class Rogue extends PlayableCharacter {
	@Override
	public void attack() {
		System.out.println(name + " performs a sneaky backstab!");
	}
}

public class Bard extends PlayableCharacter {
	@Override
	public void attack() {
		System.out.println(name + " sings a magical tune to weaken enemies!");
	}
}

class Game {
	private PlayableCharacter character;

	public Game(PlayableCharacter character) {
		this.character = character;
	}

	public void startGame() {
		character.attack();
	}
}

Here, the Game class depends only on the PlayableCharacter abstract class, not on specific implementations like Rogue or Bard. This makes it easy to inject different character types into the game, improving flexibility and reusability.
