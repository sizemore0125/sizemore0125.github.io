Interfaces

What is an Interface?

An interface in Java is a construct that defines a contract for behavior without providing implementation. It serves as a blueprint for classes, declaring methods that must be implemented by any class that chooses to "implement" the interface. Unlike classes, interfaces only specify what should be done, leaving the implementation details to the classes that implement them.

Why Use Interfaces?

Interfaces allow you to build loosely coupled, extensible, and testable applications. Remember the abstraction principle of object-oriented programming, which states that we should hide implementation details and expose only what is necessary. While classes already do this to some extent, interfaces take it further by completely separating the "what" from the "how."

When two classes interact through an interface rather than through direct references to each other, they become decoupled. The dependent class no longer relies on the implementation of the other, allowing easier modifications.

Interfaces and Abstract Classes

An interface can be compared to a contract—a promise to implement specific features. For example, if you hire an architect to design a house with a kitchen, a bathroom, and a master bedroom, the architect determines how to fulfill these requirements. In this example, the contract you use to hire the architect with all the requirements would be the interface, and the architects blueprint would be the class that implements the interface.

In contrast, an abstract class is more like a partially completed blueprint. For instance, the blueprint might include a bathroom and a master bedroom but leave the kitchen incomplete, expecting someone else to design and implement that part.

Interfaces promote loosely coupled code, while abstract classes are intended to facilitate code sharing.

Tightly Coupled Code

We have already looked at tightly coupled code with our PlayableCharacter system. Now let;s look at another example of where one class directly depends on the specific implementation details of another class:

class EmailService {
	String domain;
	int port;
	
	public EmailService(String domain, int port) {
		this.domain = domain;
		this.port = port;
	}
	
    public void sendEmail(String message) {
	    System.out.println("Connecting to server: " + domain);
        System.out.println(
	        "Email sent with message: " + message
	    );
    }
}

class Notification {
    private EmailService emailService;

	public Notification(String domain, int port) {
		this.emailService = new EmailService(domain, port);
	}

    public void notifyUser(String message) {
        emailService.sendEmail(message);
    }
}

public class Main {
    public static void main(String[] args) {
        Notification notification = new Notification();
        notification.notifyUser("Your order has been shipped!");
    }
}

This code has several problems that increase coupling. In the example code, the Notification class creates an instance of EmailService. If you want to change how notifications are sent (e.g. SMS or push notifications), then you would need to modify the Notification class directly. Additionally, if you wanted to modify the constructor or the EmailService, then you would also need to modify the Notification class. Ideally, we would want to be able to change how we send messages without changing the Notification class at all.

Creating an Interface

An interface contains only method declarations. It does not include any code or implementation. A class that implements an interface must provide concrete implementations for all of its methods unless the class is declared abstract.

Because I want my Notification class to be able to work with many different means of sending notifications, I should create a MessageService interface that other classes can implement. To do this, I use the interface keyword:

interface MessageService {
    void sendMessage(String message, String recipient);
}

Notice that we do not include the public access modifier for our method, and that is because it is redundant. We are definition the public interface for a message service, so the public access modifier is redundant.

Now, we want to modify our EmailService class to implement this interface. We can do this with the implements keyword:

class EmailService implements MessageService {
	String domain;
	int port;
	
	public EmailService(String domain, int port) {
		this.domain = domain;
		this.port = port;
	}

	@Override
    public void sendMessage(String message) {
	    System.out.println("Connecting to server: " + domain);
        System.out.println(
	        "Email sent to with message: " + message
	    );
    }
}

Finally, we will modify our Notification class's constructor to accept an object that is implemented from the MessageService interface:

class Notification {
    private MessageService messageService;

    public Notification(MessageService messageService) {
        this.messageService = messageService;
    }

    public void notifyUser(String message) {
        messageService.sendMessage(message);
    }
}

Dependency Injection

This is an example of dependency injection. Dependency Injection is a design pattern in which an object’s dependencies are provided from an external source rather than being created within the object itself. In this case, an object instantiated from a class that implements the MessageService interface is provided to the constructor of the Notification class rather than being instantiated within the class. This ensures that the Notificaiton class is only responsible for sending messages rather than making objects.

There are three kinds of dependency injection:

  1. Constructor injection
  2. Setter Injection
  3. Method Injection

In you are injecting a dependency using a constructor, this is known as constructor injection, and this is what we did above with our message service system. With constructor injection, it is the responsibility of an external source to provide our notification class with a MessageService object.

In you are injecting a dependency using a setter, this is known as setter injection. The benefit of setter injection is that you can change a dependency throughout the lifetime of the program.

Here is how we can use setter injection in our Notification class:

class Notification {
    private MessageService messageService;

    public Notification(MessageService messageService) {
        this.messageService = messageService;
    }

    public void notifyUser(String message) {
        messageService.sendMessage(message);
    }

	public void setMessageService(MessageService messageService) {
        this.messageService = messageService;
    }
}

If you are injecting a dependency into the method you use the dependency, that would be called method injection. This is useful in reducing the overhead of a class. For example, we could modify our Notification class to use method injection like this:

class Notification {
    public void notifyUser(MessageService messageService, String message) {
        messageService.sendMessage(message);
    }
}

Generally, we stick to constructor injection, and we will continue to use constructor injection for this example.

Because we have created our MessageService interface, we can create a new class that implements the MessageService interface, and that class should work with our pre-existing Notificaiton class without any issues:

class SmsService implements MessageService {
    @Override
    public void sendMessage(String message) {
        System.out.println("SMS sent to with message: " + message);
    }
}

And all of this can be used as follows:

MessageService emailService = new EmailService();
Notification notification = new Notification(emailService);
notification.notifyUser("Your order has been shipped!");

MessageService smsService = new SmsService();
Notification smsNotification = new Notification(smsService);
smsNotification.notifyUser("Your order is ready for pickup!");

Now, just as a note, these are really simple code examples. But you should consider how these patterns may help you write larger, more complicated programs without issue. You want to create reusable components of code that seamlessly and understandably work together.

Notes on Interfaces

Interfaces should be small and only have one responsibility. You don't want to create interfaces with lots of methods because that will increase coupling.

Interfaces should only be used when you want to decouple a class from its dependencies. If you have different ways of professing information that share certain features (e.g. a database), then this is a good reason to use an interface.
