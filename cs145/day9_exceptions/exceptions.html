Exceptions

What are Exceptions?

When an error occurs within a method, the method creates an object and hands it off to the runtime system. The object, called an exception object, contains information about the error, including its type and the state of the program when the error occurred. Creating an exception object and handing it to the runtime system is called throwing an exception. The Java Virtual Machine automatically throws an exception when it detects a problem.

Types of Exceptions

In Java, we have three types of exceptions:

  1. Checked Exceptions
  2. Unchecked/Runtime Exceptions
  3. Errors

Checked Exceptions

Checked exceptions are exceptions that must be either caught or declared in the method signature using the throws keyword. They are checked at compile-time, ensuring that the program handles these exceptions to prevent runtime issues.

We have see these kinds of exceptions before when we looked at how to read and write to a file. In that case, we thew an IOException in the method signature to avoid compilation errors.

For example:

import java.io.FileWriter;
import java.io.IOException;

public class WriteBook {
    public static void main(String[] args) throws IOException {
		Book book = new Book("Moby-Dick", 1851, "Herman Melville");
		FileWritter writer = new FileWriter("book_storage.txt");
		writer.write(book.getBookString() + "\n");
		// More code
		writer.close();
    }
}

Unchecked/Runtime Exceptions

Unchecked exceptions, also known as runtime exceptions, are not checked at compile-time. They occur during program execution and you are responsible for ensuring that these types of exceptions do not occur, as they happen from programming errors.

Here Are some examples of runtime exceptions:

System.out.print(1 / 0);

This will create a ArithmeticException

Book book; 
book.displayInfo();

This will create a NullPointerException.

int[] numbers = new int[3];
numbers[3] = 5;

This will create a ArrayIndexOutOfBoundsException

String x = “five”; 
double b = Double.parseDouble(x);

This will create a NumberFormatException

Errors

Errors are problems outside the application that the application cannot typically recover from. These are issues related to the JVM or the environment in which the application is running. Errors are typically unchecked. It is our responsibility as programmers to anticipate these kind of errors and prevent them from happening.

For example:

  • OutOfMemoryError (e.g., when the JVM runs out of memory)
  • StackOverflowError (e.g., caused by infinite recursion)

Exception Handling

Exception handling in Java allows a program to continue execution or handle errors gracefully when unexpected situations occur during runtime. The try and catch keywords are used to catch and handle exceptions in Java and other languages, including Python.

The basic syntax for the handling exceptions is as follows:

try { 
	// Code that may throw an exception 
} catch (ExceptionType1 e1) { 
	// Handle ExceptionType1 
}

Let's look at the example where we are writing a file:

public class WriteBook {
    public static void main(String[] args) throws IOException {
		Book book = new Book("Moby-Dick", 1851, "Herman Melville");
		FileWritter writer = new FileWriter("book_storage.txt");
		writer.write(book.getBookString() + "\n");
		// More code
		writer.close();
    }
}

Instead of modifying our method signature, we can introduce a try-catch block into the body of our method. For example:

public class WriteBook {
    public static void main(String[] args) {
		Book book = new Book("Moby-Dick", 1851, "Herman Melville");
		FileWritter writer = null;
		try {
			writer = new FileWriter("book_storage.txt");
			writer.write(book.getBookString() + "\n");
			// More code
			writer.close();
		} catch (IOException e) {
			System.out.println(e.getMessage());
		}
    }
}

*Note: we must provide a name for the variable that refers to the instance of the IOException object that is created. It is convention to call this variable e.*
*Note: The Exception object has several methods, one of which is getMessage, which will display a message about the error that occurred.*

Catching Multiple Exceptions

In the example above, we caught and IOException, but we can also catch any number of other exceptions and handle them in different ways. For example:

public class WriteBook {
    public static void main(String[] args) {
		Book book = new Book("Moby-Dick", 1851, "Herman Melville");
		FileWritter writer = null;
		try {
			writer = new FileWriter("book_storage.txt");
			writer.write(book.getBookString() + "\n");
			// More code
			writer.close();
		} catch (FileNotFoundException e) {
			System.out.println("Could not find the file!");
		} catch (IOException e) {
			System.out.println(e.getMessage());
		}
    }
}

The order of these exceptions can matter. For instance, a FileNotFoundException is a type of IOException, so if we caught the IOException first, then we would never catch a subsequent FileNotFoundException.

We can also catch two different types of exceptions in the same catch using the logical or operator (|). For example:

try {
	int[] numbers = new int[3];
	int index = Integer.parseInt(userInput);
	numbers[index] = 5;
} catch (NumberFormatException | ArrayIndexOutOfBoundsException e){
	System.out.println("Your input was bad!");
}

*Note: We are assuming that userInput is a string that a user entered into the program.*

Finally Block

The finally block is a section of code that always executes after a try-catch block, regardless of whether an exception was thrown or handled. It can be used for cleanup operations, such as closing files.

Here is an example:

import java.io.FileWriter;
import java.io.IOException;

public class WriteBook {
    public static void main(String[] args) {
		Book book = new Book("Moby-Dick", 1851, "Herman Melville");
		FileWritter writer = null;
		try {
		    writer = new FileWriter("book_storage.txt");
		    writer.write(book.getBookString() + "\n");
		    // More code
		    writer.close();
		} catch (IOException e) {
			System.out.println(e.getMessage());
		} finally {
			if (writer != null) {
				try {
					writer.close();
				} catch (IOException e) {
					System.out.println(e.getMessage());
				}
			}
		}
    }
}

In this example, we using the finally block to close the FileWriter object and free up resources in the case there is an IOException. This is a caught exception, so I must handle these sort of exceptions in my code. However, every IO operation, including writer.close(), needs to be within a try block that catches an IOException, even the one on my finally block. This makes for some ugly code.

We can make this code a little more readable using the try with resources block. This use will automatically close any open resources in the case an exception is raised. For example, we can adjust our code like this:

import java.io.FileWriter;
import java.io.IOException;

public class WriteBook {
    public static void main(String[] args) {
		Book book = new Book("Moby-Dick", 1851, "Herman Melville");
		try (FileWritter writer = new FileWriter("book_storage.txt")) {
		    writer.write(book.getBookString() + "\n");
		    // More code
		    writer.close();
		} catch (IOException e) {
			System.out.println(e.getMessage());
		} 
    }
}

Here we are instantiating our FileWriter object in parentheses after the try statement. This means that Java will automatically close this resource if an IOException is thrown.

Throws Clause

You may remember that use modified our method signature with a throws clause to avoid using a try-catch block when working with files. So, why use a try-catch block at all?

When you declare a throws clause in the method signature, you are propagating the exception to the caller of the method. It means the method itself does not handle the exception but signals that the caller must handle it or propagate it further.

If this is the way you want your program to work, then you can use the throws clause, but just know, that the buck stops somewhere, and at some point in your program you need to handle checked exceptions.

Throwing Exceptions

In Java, sometimes you will want to throw exceptions when the user of your system is interacting with it in an unintended way. For example, we implemented a Patron class that can checkout books and add those books to their inventory:

class Patron {
	int maxBooks = 5;
	Book[] inventory = new Book[maxBooks];
	int numBooksCheckedOut = 0;
	
	void checkout(Book book) {
		if (numBooksCheckedOut < maxBooks) {
			inventory[numBooksCheckedOut] = book;
			numBooksCheckedOut++;
		}
	}
}

However, if a user tries to add more books than the Patron object can hold in its inventory, then nothing happens. Instead, we should raise an exception to prevent this behavior. Let's modify our Patron class:

class Patron {
	int maxBooks = 5;
	Book[] inventory = new Book[maxBooks];
	int numBooksCheckedOut = 0;
	
	void checkout(Book book) {
		if (numBooksCheckedOut < maxBooks) {
			inventory[numBooksCheckedOut] = book;
			numBooksCheckedOut++;
		} else {
			throw new IllegalStateException("Maximum book limit reached.");
		}
	}
}

Now, when a Patron tries to checkout a book when their inventory is full, it will throw an IllegalStateException, which signals that a method has been invoked at an illegal or inappropriate time.

We should further develop our Patron class to help developers avoid instances where this exception would be thrown. For example, we should create a method called isInventoryFull that returns true when the numBooksCheckedOut is the same as maxBooks. We can also create a constructor that allows maxBooks to be set when the Patron object is created. Perhaps there are some patrons that are allowed to checkout more books than other patrons.

class Patron {
	int maxBooks;
	Book[] inventory; 
	int numBooksCheckedOut = 0;
	
	Patron(int maxBooks) {
		this.maxBooks = maxBooks;
		this.inventory = new Book[maxBooks];
	}
	
	void checkout(Book book) {
		if (numBooksCheckedOut < maxBooks) {
			inventory[numBooksCheckedOut] = book;
			numBooksCheckedOut++;
		} else {
			throw new IllegalStateException("Maximum book limit reached.");
		}
	}
	
	boolean isInventoryFull() {
		return (numBooksCheckedOut == maxBooks);
	}
}

Just as a note, we can re-throw exceptions once we caught them. This can be useful when we want to log information before an exception is thrown. For example:

try {
	int[] numbers = new int[3];
	int index = Integer.parseInt(userInput);
	numbers[index] = 5;
} catch (NumberFormatException | ArrayIndexOutOfBoundsException e){
	System.out.println("Your input was bad!");
	throw e;
}
