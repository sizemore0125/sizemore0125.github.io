Overloading and Inheritance

Let's consider our Patron class that we are developing. So far, our Patron class looks like this:

public class Patron {
	private int maxBooks;
	private Book[] inventory; 
	private int numBooksCheckedOut = 0;
	
	public Patron(int maxBooks) {
		this.maxBooks = maxBooks;
		this.inventory = new Book[maxBooks];
	}
	
	public void checkout(Book book) {
		if (numBooksCheckedOut < maxBooks) {
			inventory[numBooksCheckedOut] = book;
			numBooksCheckedOut++;
		} else {
			throw new IllegalStateException("Maximum book limit reached.");
		}
	}
	
	public boolean isInventoryFull() {
		return (numBooksCheckedOut == maxBooks);
	}
}

Let's add a removeBook method that removes a book from the inventory based on its title. The method will return true if the book is successfully found and removed, and false if no book with the specified title exists in the inventory.

public boolean removeBook(String title) { 
	boolean bookFound = false; 
	for (int i = 0; i < numBooksCheckedOut; i++) { 
		if (!bookFound && title.equalsIgnoreCase(inventory[i].getTitle())) { 
			bookFound = true; 
		} 

		// Shift books in the inventory over
		if (bookFound && i + 1 < numBooksCheckedOut) { 
			inventory[i] = inventory[i + 1]; 
		} 
	} 
	
	if (bookFound) { 
		inventory[numBooksCheckedOut - 1] = null; 
		numBooksCheckedOut--; 
	}
	
	return bookFound; 
}

This method looks good, but perhaps we want to give our users an options to remove a book based on a Book object too. In other words, we want removeBook to accept as an argument a string representing the title OR a Book object. So, how do we do this?

Method Overloading

In Python and other programming languages, you can provide functions with default arguments. For example:

def return_book(title, author=None):
	...

This means you can call this method with either title as an argument or both title and author. Additionally, since Python is not a strongly typed language, title could be any type, and the function can check the type of title and handle it accordingly based on its type.

However, Java is strongly typed and it does not allow for default arguments in its method signature. This being said, it does allow for method overloading which serves a similar purpose.

Method overloading in Java allows multiple methods in the same class to share the same name but differ in their parameter lists. To overload a method, you must vary in the number/type of parameters. However, you cannot vary the return type of a method, as this will result in a compilation error.

*Note: The return type is not considered when overloading methods!*

Let's overload our removeBook method to allow a user to either provide a title as a string or a Book object:

private boolean removeBookByTitle(String title) { 
	boolean bookFound = false; 
	for (int i = 0; i < numBooksCheckedOut; i++) { 
		if (!bookFound && title.equalsIgnoreCase(inventory[i].getTitle())) { 
			bookFound = true; 
		} 

		// Shift books in the inventory over
		if (bookFound && i + 1 < numBooksCheckedOut) { 
			inventory[i] = inventory[i + 1]; 
		} 
	} 
	
	if (bookFound) { 
		inventory[numBooksCheckedOut - 1] = null; 
		numBooksCheckedOut--; 
	}
	
	return bookFound; 
}

public boolean removeBook(String title) {
	return removeBookByTitle(title);
}

public boolean removeBook(Book book) {
	return removeBookByTitle(book.getTitle());
}

Here I extracted the logic of my old removeBook method into a private method called removeBookByTitle. Then, overload the removeBook method to allow it to accept both a String and a Book.

Method overloading simplifies complex operations by allowing the same method name to represent similar behaviors. It eliminates the need for multiple method names for similar actions, leading to cleaner code.

Here is another example of method overloading:

public class Overload{ 
	public static double getAverage(double first, double second) { 
		return (first + second) / 2.0; 
	} 
	
	public static double getAverage(double first, double second, double third) {
		return (first + second + third) / 3.0; 
	} 
	
	public static char getAverage(char first, char second){
		return (char)(((int)first + (int)second) / 2); 
	} 
}

We can call each of these methods like this:

double average1 = Overload.getAverage(40.0, 50.0); 
double average2 = Overload.getAverage(1.0, 2.0, 3.0); 
char average3 = Overload.getAverage('a', 'c');

Constructor Overloading

Currently, when we instantiate a new Patron object, we must specify the maximum number of books that a patron can check out from the library. It would be nice if we could have some default value for the maximum number of books (e.g. by default a patron can only check out 5 books).

Constructor overloading in Java allows a class to have multiple constructors with different parameter lists. This allows for flexibility in initializing objects by providing different ways to set up an instance depending on the requirements.

In constructor overloading, the constructors must have the same name as the class but differ in their parameter lists. This is the same as method overloading, but instead you are overloading the special class method, the constructor.

Let's overload our Patron class constructor:

public class Patron {
	private int maxBooks;
	private Book[] inventory; 
	private int numBooksCheckedOut = 0;
	
	public Patron(int maxBooks) {
		this.maxBooks = maxBooks;
		this.inventory = new Book[maxBooks];
	}

	public Patron() {
		this(5);
	}

Here, our Patron class has two constructors, one that takes maxBooks as an argument, and another that takes no arguments at all. Inside the body of our second constructor, we use the this keyword, which represents a specific instance of a class. In effect, the this keyword is calling our first constructor with an argument of 5.

Constructor overloading provides a way to create objects with different initial states. This is useful in scenarios where objects may require varying levels of detail or configuration during initialization.

*Note: If no constructors are defined in a class, the compiler provides a default no-argument constructor. However, once you define any constructor, the default is no longer available unless explicitly written.*

Inheritance

In OOP, Inheritance facilitates the creation of a new class from an existing class. The new class, known as the subclass or child class, and it inherits the attributes and methods of the existing class, referred to as the superclass or parent class.

This concept mimics real-world scenarios. For example, a child inherits traits from their parents.
In the context of our library, if we consider Book as a parent class, Fiction and NonFiction could be its child classes, inheriting the general characteristics of a Book. Or, if we created a PlayableCharacter class for a video game, then we could create Rogue, Wizard, or Bard subclasses, each with different characteristics. Or, we could have an Animal class, a Mammal subclass that inherits from the Animal class, and a Human subclass that inherits from the Mammal class.

Let's look at an example of inheritance using a PlayableCharacter superclass which represents common actions, and then the Rogue and Bard extend this class into specialized behavior:

public class PlayableCharacter {
    protected String name;
    protected int level;
    protected int health;

    public PlayableCharacter(String name, int level, int health) {
        this.name = name;
        this.level = level;
        this.health = health;
    }

    public void attack() {
        System.out.println(name + " attacks!");
    }
}

I first created the PlayableCharacter class, with its attributes set to protected. In Java, subclasses (such as our future Rogue and Bard classes) cannot directly access private fields in the superclass (i.e. subclasses cannot inherit private fields). By using the protected access modifier, we restrict access from external classes while still allowing subclasses to access and utilize these fields. Just as a note, it is typically considered bad practice to use protected fields, and you should stick to public and private where possible.

Now, let's make our Rogue and Bard subclasses. In Java, inheritance is achieved using the extends keyword.

public class Rogue extends PlayableCharacter {
    private int stealth;

    public Rogue(String name, int level, int health, int stealth) {
        super(name, level, health);
        this.stealth = stealth;
    }

    public void hide() {
        System.out.println(
	        name + " hides in the shadows with stealth level " + stealth + "."
	    );
    }
}

public class Bard extends PlayableCharacter {
    private String instrument;

    public Bard(String name, int level, int health, String instrument) {
        super(name, level, health);
        this.instrument = instrument;
    }

    public void playSong() {
        System.out.println(name + " plays a song on their " + instrument + ".");
    }
}

In both of these classes, we use the super keyword to call the constructor of the superclass. Notice that we give as arguments name, level, and health which maps on to the parameters of the PlayableCharacter constructor. This call to the superclass' constructor using super should be the first thing in the subclass' constructor.

Because the Rogue and the Bard extend the PlayableCharacter class, they both can use the method attack. However, they also both have their own specialized behavior hide and playSong that make them unique from each other.

We can use these two new classes like this:

Rogue rogue = new Rogue("Aragorn", 5, 80, 100); 
Bard bard = new Bard("Tom Bombadil", 10, 120, "Lyre");

rogue.attack();
rogue.hide()

bard.attack();
bard.hide();

The Object Class

In Java, the Object class is the root of the class hierarchy. Every class in Java, either directly or indirectly, inherits from the Object class. This means that all Java objects have access to the methods defined in the Object class.

We saw that our Bard class extends (or inherits from) the PlayableCharacter class, and the PlayableCharacter class extends (or inherits from) the Object class. We can image that the class definition looks like this:

public class PlayableCharacter extends Object {

However, the Java compiler adds this for us. This is the reason that every class that we create has some additional methods, such as comparison, hashing, and string representation.

For example, the method hashCode returns an integer hash code for the object. We can look at it here:

PlayableCharacter logan = new PlayableCharacter("Logan", 9000, -1);
System.out.println(logan.hashCode());

Which will display something like this:

705927765

This is an integer representation calculated from the address of the logan object in memory. This hash code can be used to compare objects for equality. For example, I will declare a new PlayableCharacter object and set it to the value of logan:

PlayableCharacter logan = new PlayableCharacter("Logan", 9000, -1);
PlayableCharacter logan2 = logan;
System.out.println(logan.hashCode());
System.out.println(logan2.hashCode());

We know how reference types work in Java, so it should be no surprise that this will display the same hash code:

705927765
705927765

Another method that all objects in Java have is the equals method. The equals method simply compares the current object with another object to determine equality. By default, it checks whether two objects point to the same memory location.

For example:

PlayableCharacter logan = new PlayableCharacter("Logan", 9000, -1);
PlayableCharacter logan2 = logan;
System.out.println(logan.equals(logan2));

PlayableCharacter logan3 = new PlayableCharacter("Logan", 9000, -1);
System.out.println(logan.equals(logan3));

This will display:

true
false

Another method that is implemented by the Object class is the toString method. This method returns a string representation of the object. By default, it prints the class name followed by the object’s hash code in hexadecimal.

For example:

PlayableCharacter logan = new PlayableCharacter("Logan", 9000, -1);
System.out.println(logan.toString());

This will display:

PlayableCharacter@2a139a55

Perhaps these methods do not work as we would want them to. For example, we may want the equals method to compare all of the fields of our class. Or, perhaps we want the toString method to display the name and level of our PlayableCharacter class. In this case, we would have to override these methods.

Overriding Methods

Method Overriding is a feature that allows a subclass to provide a different implementation for a method that is already defined in its superclass.

*Note: Method overriding differs from method overloading. Overriding involves replacing a method defined in a superclass with a specific implementation in a subclass. Overloading provides multiple implementations of the same method within the same class, distinguished by different parameter lists.*

Let's say that we want to override a the toString method defined in the Object class for our PlayableCharacter class. We can do this using the @Override annotation. An annotation is a label that we can attach to a class member, and this gives extra information to the Java compiler. 

public class PlayableCharacter {
    protected String name;
    protected int level;
    protected int health;

    public PlayableCharacter(String name, int level, int health) {
        this.name = name;
        this.level = level;
        this.health = health;
    }

	@Override
	public String toString() {
		return "Name: " + name + "\nLevel: " + level + "\nHealth: " + health + "\n"
	}

    public void attack() {
        System.out.println(name + " attacks!");
    }
}

Now, when we use this code:

PlayableCharacter logan = new PlayableCharacter("Logan", 9000, -1);
System.out.println(logan.toString());

This will display:

Name: Logan
Level: 9000
Health: -1

Just as a note, we do not explicitly need to call the toString method in this case because the println method automatically calls the toString method on any object we pass in as an argument.

So, this code:

PlayableCharacter logan = new PlayableCharacter("Logan", 9000, -1);
System.out.println(logan);

Will display:

Name: Logan
Level: 9000
Health: -1

Currently, our PlayableCharacter implements the method attack. However, it may be the case that we want our two subclasses to also implement an attack method that has unique behavior. So, let's do that:

public class Rogue extends PlayableCharacter {
    private int stealth;

    public Rogue(String name, int level, int health, int stealth) {
        super(name, level, health);
        this.stealth = stealth;
    }

    public void hide() {
        System.out.println(
	        name + " hides in the shadows with stealth level " + stealth + "."
	    );
    }

	@Override
	public void attack() {
		System.out.println(name + " performs a sneaky backstab!");
	}
}

public class Bard extends PlayableCharacter {
    private String instrument;

    public Bard(String name, int level, int health, String instrument) {
        super(name, level, health);
        this.instrument = instrument;
    }

    public void playSong() {
        System.out.println(name + " plays a song on their " + instrument + ".");
    }

	@Override
	public void attack() {
		System.out.println(name + " sings a magical tune to weaken enemies!");
	}
}
